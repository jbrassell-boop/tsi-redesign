1. Architecture Overview

Application Type

Total Scope, Inc. is a single-page application (SPA) built with React and React Router v6. It is a medical device repair management platform used by service technicians, administrators, and client-facing staff to track scopes (endoscopes), repair workflows, tasks, loaners, inventory, acquisitions, quality control, and financials.

API Architecture

The frontend communicates with a separate REST API hosted at totalscopetestapi.mol-tech.com. Key endpoint patterns discovered via performance.getEntriesByType('resource'):

  • /api/ScopeModel/GetScopeModelDropDown — Returns all scope model options (1,264 items, all loaded at once)
  • /api/ScopeModel/GetScopeModelById/{id} — Returns full model record
  • /api/RepairItem/GetRepairItemDropDown — Returns 481 repair item options
  • /api/Manufacturer/GetManufacturerDropDown — Returns manufacturer list
  • /api/Task/GetTaskList — Returns task rows
  • /api/Task/SaveTask — POST to save task
Architecture Warning: All dropdown data is fetched and loaded into memory on page load with no virtualization or pagination on the client side. At 1,264 items in Scope Model alone, this creates unnecessary memory pressure and slow initial renders.

Authentication

The application uses session-based authentication. Navigating to a protected route while unauthenticated redirects to /login. The login screen accepts email + password credentials with an optional "Remember Me" checkbox and a "Forgot Password?" link. No SSO or OAuth observed.

7. Modal & Panel Patterns

The application uses two distinct overlay patterns. Understanding both is critical for recreating the UI:

Pattern A: Right-Side Slide Panel

CSS class: .right-model-section. Width: ~50% of viewport. Slides in from the right edge. Used for: Add Task, Update Task, Add Flag. Characteristics:

  • Header bar with icon + title (e.g. "Add Task") + X close button in top-right
  • Header background matches the app's primary blue
  • Content area is scrollable independently of the background page
  • Background page content is still visible/accessible on the left
  • No overlay backdrop — the page behind remains fully interactive
  • Save + Cancel buttons at the bottom (Save = dark blue, Cancel = dark gray)

Pattern B: Centered Dialog Modal

CSS class: .center-model-section. Width: ~500-600px centered. Used for: Add Repair Item, Manufacturer gear, Video Image gear, Import/Export. Characteristics:

  • Dark overlay backdrop covers the full page
  • Dialog centered horizontally and vertically
  • Same blue header with icon + title + X close button
  • Save + Cancel buttons at the bottom
  • Clicking outside the dialog (on the backdrop) may or may not close — not confirmed

9. Architecture Issues & Improvement Recommendations

This section is written specifically for Claude Code and the development team as a redesign specification input. Each issue is paired with a concrete recommendation.

Issue A: Mixed Component Library Stack

Current state: The app uses at least 4 different input component libraries simultaneously on the same page: React-Select, PrimeReact, react-datepicker, and Bootstrap native inputs. Each has different styling, keyboard nav behavior, accessibility support, and CSS class naming.

Recommendation: Pick one component library as the foundation. Recommended options: (1) shadcn/ui + Radix UI — headless, fully accessible, easy to style with Tailwind; (2) Mantine — full-featured, consistent; (3) Ant Design — enterprise-focused, high component coverage. All React-Select, PrimeReact, and react-datepicker usage should be replaced with the chosen library's equivalents.

Issue B: No Virtualization on Large Dropdowns

Current state: 1,264 scope model options, 481 repair item options, and 321 scope categories all loaded into unvirtualized dropdowns at page load.

Recommendation: Implement server-side filtering + async loading: use react-select/async with debounced API search. Only fetch matching options as the user types. Cache frequently accessed items in a React Query or SWR cache.

Issue C: No Pagination on Tables

Current state: Dashboard Scopes tab shows 202+ rows with zero pagination. This will grow over time and eventually cause the page to become unusable.

Recommendation: Implement server-side pagination with configurable page size (25/50/100). Use cursor-based pagination for better performance at scale. Consider TanStack Table (React Table v8) for the table layer — it handles sorting, filtering, and pagination in a headless, composable way.

Issue D: No Column Sorting

Current state: No sortable columns anywhere in the application.

Recommendation: TanStack Table's getSortedRowModel() handles client-side sort. For server-side sort, pass sort params to the API. At minimum, Date In, Client, Repair Status, and Priority should be sortable on Dashboard/Scopes.

Issue E: ScopeModel Has No List View

Current state: Users must know the exact model name to navigate to it via the dropdown. There is no browse/search interface.

Recommendation: Add a standard list view as the default for /ScopeModel, with a data table showing: Description, Manufacturer, Model Category, Instrument Type, Active status, Contract Cost. Clicking a row navigates to /ScopeModel/{id} in edit view. The current toolbar-based editor becomes the record detail page.

Issue F: Non-Bookmarkable Record URLs

Current state: Selecting a scope model via dropdown does not update the URL. /ScopeModel always stays as-is regardless of which model is loaded. Same issue likely exists on other pages with dropdown-driven record selection.

Recommendation: Update the URL on record selection: /ScopeModel/GIF-H190 or /ScopeModel?model=12345. This enables bookmarking, browser back/forward navigation, and sharing direct links to specific records.

Issue G: Iconify Runtime CDN Dependency

Current state: Some icons are loaded at runtime from api.iconify.design (external CDN). If this CDN is unavailable, icons fail silently and parts of the UI become icon-less.

Recommendation: Bundle all icons as static SVG imports or use a single icon library (Lucide React or Heroicons) installed as an npm package. Eliminate all external CDN icon dependencies.

Issue H: API Data / Label Mismatches

Current state: The "Notes" column in the Tasks table maps to psCustomerMessage at the API level. Field naming conventions appear inconsistent between the UI and the API.

Recommendation: Create a typed API schema layer (e.g. using Zod or TypeScript interfaces generated from the OpenAPI spec). UI components should consume named, typed fields — not raw API field names.

Issue I: Test Data in Production Schema Options

Current state: Video Image dropdown includes "a" as a valid option. Model Category dropdown includes "New Cat", "test", "TBD". These are clearly test entries that contaminate the reference data.

Recommendation: Add server-side validation to reject single-character or common test string values (case-insensitive "test", "TBD") in admin dropdowns. Implement a data cleanup migration to remove existing invalid entries.

Issue J: Bootstrap via CDN (Not NPM)

Current state: Bootstrap 5.3 is loaded via CDN <link> tag, not installed as an npm package. This prevents tree-shaking, makes upgrades manual, and can cause CSS conflicts with React component libraries.

Recommendation: If keeping Bootstrap, install via npm and import only the needed modules. Better yet, replace Bootstrap with Tailwind CSS for utility-first styling that works cleanly with any component library.
